# 学生成绩在线发布系统 - 项目协作规范文档

> 适用团队：4 人开发小组（A / B / C / D）  
> 技术栈：JSP + Servlet + JDBC + MySQL  
> 使用方式：每人打开自己的看板 + 阅读规则速查表 + 了解冲突点即可

## 第四部分：关键冲突点提醒

### ⚠️ 冲突点 1：A 同学改表结构导致 B/C/D 全部返工

**是什么**：A 同学在建表后发现字段设计不合理，修改了表名或字段名/类型

**为什么容易出问题**：B/C/D 已经按照旧表结构写好了存储过程、触发器、视图、索引，表结构一改全部报错

**怎么预防**：
- **A 同学负责**：建表前先在组内评审 DDL，确认无误后再发布；发布后锁定表结构，非必要不改
- **B/C/D 负责**：如果发现表结构有问题，第一时间反馈给 A，不要自己先写 SQL 再说

---

### ⚠️ 冲突点 2：C 同学的触发器和 B 同学的存储过程争抢 scores 表写权限

**是什么**：C 同学的触发器在 scores 表变更时自动写入 audit_log，而 B 同学的存储过程也可能批量更新 scores 表

**为什么容易出问题**：如果 B 同学的存储过程批量 INSERT/UPDATE scores，每次操作都会触发 C 的触发器写入 audit_log，可能导致大量无用日志或性能问题

**怎么预防**：
- **B 同学负责**：存储过程中涉及 scores 写操作时，提前告知 C 同学，确认触发器是否需要特殊处理
- **C 同学负责**：触发器中加条件判断，如果是批量导入操作可以跳过审计日志（通过会话变量控制）
- **A 同学负责**：在集成时统一协调两者的执行顺序

---

### ⚠️ 冲突点 3：D 同学的视图依赖 C 同学的索引，但 C 还没建好

**是什么**：D 同学写好了视图 SQL，但查询很慢，因为 C 同学的索引还没创建

**为什么容易出问题**：D 同学可能先用没索引的表测试视图，觉得没问题就交付了，但上线后数据量一大就卡死

**怎么预防**：
- **D 同学负责**：写视图前先问 C 同学要索引脚本，在本地建好索引后再测试视图性能
- **C 同学负责**：索引脚本写完后优先交付给 D 同学，不要等最后一起交
- **两人约定**：C 的索引脚本完成后，D 再开始写视图的最终版本

---

### ⚠️ 冲突点 4：A 同学集成时 SQL 执行顺序出错

**是什么**：A 同学拿到所有人的 SQL 脚本后，执行顺序不对导致报错（比如先执行视图脚本，但表还没建）

**为什么容易出问题**：4 个人各自独立写 SQL，没人关心别人的脚本依赖什么，集成时容易乱

**怎么预防**：
- **A 同学负责**：按以下固定顺序集成执行：
  1. 建表 DDL（A 自己的）
  2. 索引脚本（C 同学的）
  3. 触发器脚本（C 同学的）
  4. 存储过程脚本（B 同学的）
  5. 视图脚本（D 同学的）
  6. 权限脚本（D 同学的）
  7. 备份脚本（D 同学的）
- **B/C/D 负责**：每个 SQL 脚本开头加注释说明依赖项，方便 A 同学排查问题
