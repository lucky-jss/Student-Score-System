

# 审计日志触发器 + 索引优化 交付文档
> **所属项目**：学生成绩在线发布系统
> **模块分工**：C模块 - 数据库自动化维护与性能优化
> **环境依赖**：MySQL 8.0+ / InnoDB 存储引擎
> **前置条件**：项目 `init.sql` 已完成基础表结构与测试数据初始化

---

## 一、模块概述
本模块为学生成绩在线发布系统的**数据库增强功能模块**，从「数据安全审计」和「查询性能优化」两个维度完善数据库能力，全部逻辑在数据库层实现，对业务代码无侵入。

### 1.1 审计日志触发器（核心功能）
- 对成绩核心表 `scores` 的 **INSERT（新增）、UPDATE（修改）、DELETE（删除）** 三类写操作进行自动捕获
- 操作日志结构化存入 `audit_log` 审计表，实现成绩数据变更全流程可追溯
- 全程由MySQL触发器自动触发，无需Java后端编写额外逻辑，降低业务层复杂度

### 1.2 索引优化（性能功能）
- 基于系统高频业务查询场景，针对性设计单列索引与联合索引
- 覆盖学生端成绩查询、教师端课程成绩统计、学期维度成绩分析等核心场景
- 通过 `EXPLAIN` 执行计划验证索引生效，显著提升大数据量下的查询效率

## 三、审计日志触发器设计
### 3.1 设计思路
1. **触发时机**：采用 `AFTER` 触发，确保业务操作执行成功后再记录日志，避免操作失败产生无效日志
2. **数据捕获**：通过MySQL内置的 `NEW` / `OLD` 关键字，精准获取数据变更前后的完整字段值
3. **存储格式**：使用 `JSON_OBJECT` 函数将变更数据封装为JSON格式，结构化存储，便于后续扩展、解析与分析
4. **自动补全**：自动填充操作人、操作时间、操作IP等审计字段，无需业务层传参

### 3.2 审计日志表结构（audit_log）
| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| `id` | INT | 主键、自增 | 日志唯一标识 |
| `user_id` | INT | 非空 | 操作用户ID，关联users表 |
| `action` | VARCHAR(20) | 非空 | 操作类型：INSERT / UPDATE / DELETE |
| `table_name` | VARCHAR(50) | 非空 | 被操作的表名，此处固定为scores |
| `record_id` | INT | 非空 | 被操作的成绩记录ID |
| `old_data` | TEXT | 可空 | 修改前的完整数据（JSON格式），新增操作该字段为NULL |
| `new_data` | TEXT | 可空 | 修改后的完整数据（JSON格式），删除操作该字段为NULL |
| `ip_address` | VARCHAR(45) | 可空 | 操作端IP地址 |
| `created_at` | DATETIME | 非空、默认当前时间 | 操作发生时间 |

### 3.3 触发器实现详情
#### 3.3.1 新增操作触发器（trg_scores_insert）
- **触发时机**：成绩数据插入后
- **功能**：自动记录新增的成绩信息，捕获新插入的完整数据

#### 3.3.2 修改操作触发器（trg_scores_update）
- **触发时机**：成绩数据更新后
- **功能**：同时记录修改前后的完整数据，支持变更对比追溯

#### 3.3.3 删除操作触发器（trg_scores_delete）
- **触发时机**：成绩数据删除后
- **功能**：留存被删除数据的完整快照，防止数据无痕丢失

### 3.4 设计价值
- 实现数据变更全链路自动记录，满足系统审计要求
- 数据库层实现，对业务代码零侵入，无需修改Java层逻辑
- 保障数据可追溯性，提升系统数据安全性
- 符合教学管理系统的数据安全规范

## 四、索引优化设计
### 4.1 设计原则
1. **场景驱动**：基于系统真实高频查询场景设计，不盲目建索引
2. **最左前缀**：联合索引遵循最左前缀原则，最大化索引复用率
3. **读写平衡**：控制索引总数量，优先优化WHERE、JOIN关联字段，避免过度索引影响写性能
4. **可验证**：所有索引均可通过 `EXPLAIN` 执行计划验证生效效果

### 4.2 索引设计方案
#### scores 成绩表（核心优化对象）
| 索引名称 | 索引字段 | 索引类型 | 适配业务场景 |
|----------|----------|----------|--------------|
| `idx_student_semester` | `student_id, semester_id` | 联合索引 | 学生按学期查询个人所有成绩 |
| `idx_subject_semester` | `subject_id, semester_id` | 联合索引 | 教师按学期查询单课程所有学生成绩 |
| `idx_score` | `score` | 单列索引 | 成绩分段统计、排名计算场景 |

#### students 学生表
| 索引名称 | 索引字段 | 索引类型 | 适配业务场景 |
|----------|----------|----------|--------------|
| `idx_student_no` | `student_no` | 单列索引 | 学号快速查询学生信息 |
| `idx_class_id` | `class_id` | 单列索引 | 按班级批量查询学生 |
| `idx_status_deleted` | `status, is_deleted` | 联合索引 | 学生状态筛选、逻辑删除过滤 |

#### users 用户表
| 索引名称 | 索引字段 | 索引类型 | 适配业务场景 |
|----------|----------|----------|--------------|
| `idx_username` | `username` | 单列索引 | 登录账号快速校验 |
| `idx_department_id` | `department_id` | 单列索引 | 按院系筛选用户 |

## 五、功能验证方法
### 5.1 触发器功能验证
#### （1）新增触发器测试
```sql
INSERT INTO scores (student_id, subject_id, semester_id, score, grade, recorded_by)
VALUES (2, 3, 1, 95, 'A', 2);

-- 验证：查询审计表，确认生成INSERT类型日志
SELECT * FROM audit_log WHERE action = 'INSERT' ORDER BY id DESC LIMIT 1;
```
预期结果：自动生成1条审计记录，`new_data` 字段包含插入的完整成绩数据，`old_data` 为NULL。

#### （2）修改触发器测试
```sql
UPDATE scores
SET score = 97, grade = 'A+'
WHERE student_id = 1 AND subject_id = 1 AND semester_id = 1;

-- 验证：查询审计表，确认生成UPDATE类型日志
SELECT * FROM audit_log WHERE action = 'UPDATE' ORDER BY id DESC LIMIT 1;
```
预期结果：自动生成1条审计记录，`old_data` 和 `new_data` 分别存储修改前后的完整数据。

#### （3）删除触发器测试
```sql
DELETE FROM scores
WHERE student_id = 2 AND subject_id = 3 AND semester_id = 1;

-- 验证：查询审计表，确认生成DELETE类型日志
SELECT * FROM audit_log WHERE action = 'DELETE' ORDER BY id DESC LIMIT 1;
```
预期结果：自动生成1条审计记录，`old_data` 字段存储被删除的完整成绩数据，`new_data` 为NULL。

### 5.2 索引有效性验证
使用 `EXPLAIN` 执行计划验证索引是否生效，示例语句：
```sql
EXPLAIN SELECT * FROM scores WHERE student_id = 1 AND semester_id = 1;
```

#### 执行计划核心字段说明
| 字段 | 含义 | 生效判定标准 |
|------|------|--------------|
| `type` | 查询访问类型 | 值为 `ref` / `range` 表示索引查询，`ALL` 为全表扫描 |
| `key` | 实际使用的索引 | 显示对应创建的索引名称，证明索引被命中 |
| `rows` | 预估扫描行数 | 数值远小于表总行数，说明索引过滤有效 |
| `Extra` | 额外信息 | 出现 `Using index condition` 表示索引条件下推优化生效 |

### 5.3 性能对比
| 状态 | 查询方式 | 性能表现 |
|------|----------|----------|
| 无索引 | 全表扫描 | 数据量增大后查询速度线性下降，万级数据延迟明显 |
| 有索引 | 索引查找（ref） | 查询速度稳定，大数据量下性能提升数十倍，支持高并发查询 |
